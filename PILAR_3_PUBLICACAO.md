# Pillar 3 — Publishing (Publicação)

> Reference spec for the Publishing pillar. It describes **what** a complete
> version of this pillar is responsible for and the capabilities it should
> expose. Product decisions here are already made (the metadata engine, the
> thumbnail design); what is left to you is purely technical — how to composite
> the thumbnail text, how the config schema is shaped, where files live, how
> scheduling is wired. Choose those consistent with the project's conventions.
>
> The same two principles from the earlier specs run through everything below:
> **efficiency** (uploads are quota-expensive, so never spend a quota-heavy call
> twice and never re-do metadata or a thumbnail that already exists) and
> **configurability** (every destination, template, toggle, and default is
> panel-editable, never hardcoded). Failures degrade instead of crashing, matching
> the posture used across the codebase.

## Responsibility (one sentence)

Publishing takes the finished video file and its sidecar and gets it live on its
destination platform(s): it produces the platform-facing metadata and thumbnail,
sets visibility and disclosure, uploads at the right time, and records where the
video went. It owns everything from "a finished video file" to "the video is live,
with its platform IDs and URLs recorded" — and nothing about choosing the theme,
making the video, or measuring how it did.

## Input / output model

**Input:** the finished video file plus its sidecar record from Generation (the
sidecar carries the script and theme, which metadata and thumbnail text read).
**Output:** the video live on each enabled destination, recorded with its
per-platform video ID and URL and the metadata it was published with. That record
is the seam the Feedback pillar later reads to pull analytics. Publishing does not
make the video and does not measure it; the finished file is its input and the
published-record is its output.

## The flow this pillar is responsible for

```
finished video + sidecar → metadata (Groq) → thumbnail (bg image + Groq text) → [disclosure / visibility / review gate] → upload per enabled destination (scheduled or immediate) → recorded as published (IDs, URLs)
```

## Efficiency principle

A platform upload is expensive in quota, not dollars: on YouTube a single upload
consumes a large share of the standard daily quota (on the order of 1,600 of the
default 10,000 units), so only a handful fit per day per credential unless quota is
raised. Spend that expensive call carefully: never upload the same video twice
(idempotency), track quota so a run stops before it exhausts the day's budget, and
never regenerate metadata or a thumbnail that already exists and still validates.

## Configurability principle

Nothing about how a video is published is hardcoded. Which destinations are
enabled, the credential each uses, the metadata templates and tone, the thumbnail
text font and color and position and outline and the background image source, the
publish timing, the default visibility and disclosure, and the review-gate mode —
all read from configuration and editable from the admin panel, with sensible
defaults so a video publishes correctly with no tuning. Publishing defines the
config schema; the Control pillar renders the forms over it. See "What must be
panel-configurable" below.

## Capabilities of a complete Publishing pillar

### 1. Metadata optimization (Groq)
The theme is not the title. A dedicated step turns the theme and script (from the
sidecar) into a platform-facing title, description, and tags optimized for
click-through and search — not the raw theme string pasted into the title field.
The engine is **Groq**, the same LLM already used for the script, reusing the
project's existing integration and credentials; no new provider is added. This step
is always on; what is configurable is the templates, the tone, and the tag
strategy. This is the single highest-leverage lever in the pillar.

### 2. Thumbnail (background image + Groq text)
For long-form video the thumbnail is the biggest click-through lever there is, so
the pillar builds one rather than letting the platform auto-pick. The design is
fixed; the composition is yours to implement:

- **Background image**, whose *source is a panel setting*: either **AI-generated**
  (through the Together/FLUX provider already in the project) or from an **image
  bank** (through Pexels, already in the project). For the AI mode the image
  **prompt** is generated by **Groq**; for the bank mode the **search term** is
  generated by **Groq**.
- **Overlay text**, generated by **Groq** from the theme/script.
- The text is composited over the background with **panel-configurable font,
  color, position, and outline**, and the whole thumbnail can be turned off per
  type (it matters far less for short-form).

No new provider or credential is introduced — background image and text reuse the
FLUX/Pexels/Groq integrations already present. Only the compositing mechanism (how
the text is drawn onto the image) is a technical choice for you.

### 3. Pluggable destinations with credential isolation
Like Discovery's sources and Generation's providers, a "destination" is a role any
platform can fill behind a common contract, so a new platform is added by
implementing the contract, not by rewiring the pillar. Each destination has its own
enable/disable toggle and its own isolated credential, so one channel's quota or
auth never affects another. The pillar handles the credential lifecycle: an expired
or soon-to-expire credential (the 7-day refresh-token expiry already hit in Testing
mode is the canonical example) is detected and surfaced clearly, never a silent
failure. YouTube is the destination wired now, reusing the project's existing
credentials; TikTok, Reels, and Kwai are left as a clean seam behind the same
contract.

### 4. Scheduled / decoupled publish
When a video goes live is decoupled from when generation finished. The pillar can
upload a video as private/scheduled and set its go-live time to a configured optimal
window per type, rather than publishing whenever the file happened to be ready.
**Default is immediate publish**; scheduling is an available, configurable mode
(there is no optimal-time data yet to justify turning it on by default).

### 5. Disclosure, visibility & review gating
Publishing is where the upload's flags are actually set: the visibility (**default
public**), the audience setting, and the synthetic-media disclosure flag that the
platform has required since January 2026 (**default on** — Compliance owns the
judgment of when disclosure applies, but Publishing is where the flag is set). It
also enforces the review gate: **default auto-publish** (the channel is autonomous),
with review-first available as a per-type toggle in which a finished video waits for
human approval before going live — this is what the existing per-type publish toggle
becomes.

### 6. Idempotency / no double-publishing
A video is never uploaded twice. The pillar records what was published (per-platform
ID and URL) and a retried publish after a partial failure resumes and reconciles
rather than re-uploading, honoring the existing one-run-per-type invariant. This
protects both the channel (no duplicates) and the quota (no wasted uploads).

### 7. Quota & cost awareness
Because uploads are quota-heavy, each destination tracks the quota it consumes, and
the pillar enforces a configurable daily cap (max uploads per day) per credential,
set conservatively to stay under the standard quota by default, deferring to the
next day rather than exhausting it. Any paid step (Groq metadata, AI-generated
thumbnail background) is counted too. The numbers feed Operations and the panel.

### 8. Observability
For each publish: what was uploaded where, with what metadata and thumbnail, the
resulting platform ID and URL, the visibility and disclosure that were set, whether
it succeeded, and the quota it used. Recording this is Publishing's job; surfacing it
is Control's. A failed destination degrades on its own — the others still proceed,
and the failure is recorded rather than crashing the run.

## What must be panel-configurable (not hardcoded)

Per destination: an enable/disable toggle, the credential it uses, and its upload
defaults (category, language, playlist). For metadata: the templates, the tone, and
the tag strategy (the engine is Groq, fixed). For the thumbnail: the background
image source (AI-generated vs image bank), the text font, the text color, the text
position, the text outline, and on/off per type. For timing: immediate vs scheduled,
and the optimal-time schedule per type. For visibility: the default privacy, the
audience setting, and the default synthetic-media disclosure flag. For the review
gate: auto-publish vs review-first per type. For quota: the daily upload cap per
credential and the over-cap behavior. Everything ships with a sensible default.

## Hand-off contract

Publishing's only input is the finished video file plus its sidecar from Generation;
its only output to the rest of the system is the published-record (per-platform IDs,
URLs, and the metadata used), which Feedback later reads. Neither boundary changes
shape in a way that breaks the neighbor. Credentials and providers already in use
(YouTube, Groq, FLUX, Pexels) are reused; no new credential setup is performed here.

## Integrations to fix (part of this work, not a follow-up)

The current YouTube upload path — title, description, hashtags, privacy, per-type
OAuth — moves onto this model without changing behavior for a default
configuration: the upload call is wrapped behind the destination contract, the Groq
metadata step and the thumbnail step are inserted ahead of it, the handoff from
Generation becomes "finished file plus sidecar," and the run-history records gain
the published-record (IDs, URLs, quota used). The existing per-type publish toggle
becomes the review-gate setting. The test suite stays green and the default-config
behavior stays equivalent to today's.

## Explicitly out of scope for this pillar

Choosing the theme (Discovery), making the video (Generation), the panel forms that
edit these settings (Control), pulling analytics and measuring performance
(Feedback), scheduling and running the pipeline and firing alerts (Operations), and
the policy judgment of what must be disclosed or is allowed (Compliance — Publishing
only sets the flags it decides). Publishing ships the video; it doesn't choose it,
make it, run itself, or grade it.
